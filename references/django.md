# Django conventions

## Models
- **Fat models, thin views.** Business logic and invariants live on models,
  managers, and `QuerySet` methods — not in views.
- Every model defines `__str__`. Set `Meta.ordering` explicitly if order matters
  anywhere (no relying on DB default order).
- Use explicit field types and constraints; prefer DB-level constraints
  (`UniqueConstraint`, `CheckConstraint`) over app-only validation for invariants.
- Custom managers/`QuerySet`s for reusable queries; don't scatter the same filter
  across the codebase.

## Queries
- **Never query in a loop.** Use `select_related` (FK/one-to-one) and
  `prefetch_related` (M2M/reverse) to avoid N+1.
- Don't load more than you need: `.only()`/`.defer()`/`.values()` where it counts;
  `.exists()` instead of `len(qs)`; `.count()` instead of `len(list(qs))`.
- Wrap multi-write operations in `transaction.atomic`.

## Views
- Keep views thin: validate, delegate to model/service code, return a response.
- Enforce authorization explicitly; never trust client-supplied object ids without
  an ownership/permission check.

## Migrations
- Migrations are reviewed code: no autogenerated cruft committed blindly.
- Separate schema changes from data migrations; make data migrations reversible or
  explicitly document why not.
- Name migrations meaningfully. Avoid destructive operations without a deliberate,
  noted decision.

## Security & config
- Never hardcode secrets or `SECRET_KEY`; read from environment (and document the
  vars in `.env.example`).
- `DEBUG` off by default; `ALLOWED_HOSTS` set from config.
- Use the ORM / parameterised queries; never string-format user input into SQL.
